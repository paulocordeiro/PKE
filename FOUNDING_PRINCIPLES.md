# PKE Founding Principles

These principles constrain the evolution of the **Project Knowledge Engine (PKE)**. They are intentionally brief and should change rarely.

## 1. Specification first

PKE is an open specification before it is an implementation. Reference software must follow the specification rather than define it implicitly.

## 2. Knowledge over context

Knowledge is the durable asset. Context is a temporary delivery mechanism assembled for a specific task.

## 3. AI-independent

PKE must not depend on a particular model vendor, agent framework, IDE, programming language, or storage technology.

## 4. Minimal conceptual surface

A new concept may enter the specification only when existing concepts cannot solve the problem clearly. Naming something creates a permanent maintenance obligation.

## 5. Evidence-driven decisions

Important decisions must state the problem, supporting evidence, alternatives, and trade-offs. Attractive ideas remain hypotheses until evidence justifies standardization.

## 6. Knowledge Budget

Every artifact, concept, dependency, and requirement consumes a **Knowledge Budget**.

A change should be accepted only when the complexity it removes or the capability it enables is greater than its lasting cost of comprehension and maintenance.

The budget is not primarily a token or page limit. It is a constraint on how much knowledge a person or agent must understand to make a correct decision or complete a task.

## 7. Small, reversible steps

The project should reduce uncertainty through focused increments. Premature detail and speculative scope should be deferred.

## 8. Dogfooding

PKE should be developed using the principles it specifies. Difficulty locating decisions, understanding concepts, or assembling sufficient context is direct feedback about the specification.

## 9. Simplicity wins

Prefer the simplest design that preserves required expressiveness and interoperability.

## 10. Semantic parity across languages

English and Brazilian Portuguese documentation must preserve the same meaning. Translation must not create separate specifications.

---

**Guiding statement:** Knowledge is an asset. Context is a delivery mechanism.
